## 4. Failure resolution endpoint & backfill

- [x] 4.1 Rewrote `app/api/admin/td-sync/failures/[id]/resolve/route.ts` with an `action` field (`link` | `create` | `configure` | `dismiss` | `ignore`) instead of the old `{workerId, dismiss, ignore}` shape. `link` stays UNMATCHED-only and unchanged (no backfill — a linked worker may still not be configured). `create` (UNMATCHED → new worker) and `configure` (NEEDS_SETUP/MISSING_RATE → existing worker) both validate rate/currency/paymentType/payment-account via a shared `parseProvisionFields` and call `provisionWorker`.
- [x] 4.2 `create`/`configure` sweep unresolved sibling failures via `OR: [{ workerId }, { workerId: null, tdEmail: {equals, mode:"insensitive"} }]` — covers both "worker already existed for earlier months" and "this is the first month, worker didn't exist until this resolve".
- [x] 4.3 `backfillInvoices()` in the resolve route calls `buildTdInvoiceData` (#3.5) per swept failure using its stored `quantity`/`billingMonth`, producing historically-correct `invoiceDate`/`dueDate`.
- [x] 4.4 Worker provisioning + failure sweep + invoice backfills run inside one `db.$transaction`. One deliberate exception, matching existing precedent (`app/api/invoices/route.ts` does the same): invoice-number reservation (`generateInvoiceNumber`/`reserveInvoiceNumbers`) happens *before* the transaction, since `lib/invoice-number.ts` isn't tx-aware — a rolled-back resolve burns reserved numbers, same accepted trade-off as elsewhere in this codebase. `@@unique([workerId, billingMonth, supplementNo])` P2002 is caught per-invoice inside `backfillInvoices`, so a retried/duplicate resolve is a no-op there.
- [x] 4.5 `buildTdInvoiceData` hard-codes `vatInclusive: true` unconditionally (no parameter exists to override it) — covers the resolve path. The manual-add path (group 5) never creates an invoice at all (pure-Manual workers self-report every line item), so there's no second "invoice path" for this to apply to.
