## Context

`td-sync-worker-onboarding` closes the actual security gap (an unmatched `User` can no longer obtain a working `Worker` profile via `GET /api/profile`). This change is optional hardening on top: once the known migration window (the ~260 pre-existing CSV-imported workers claiming their accounts) is over, there's no remaining legitimate reason for `POST /api/auth/sign-up/email` to accept arbitrary emails, so Admin wants a way to turn it off — as a runtime setting, not a deploy.

## Goals / Non-Goals

**Goals:**
- Give Admin a single toggle, changeable without a deploy, that closes self-service account creation to only pending-matched emails.
- Preserve the first-user bootstrap path (empty `User` table) regardless of the toggle's value, so the app can never be locked out of having an admin.
- Give Admin visibility into how many pending workers are still unclaimed, so the decision to flip the toggle is informed rather than a guess.
- Give a worker created via the "Add worker" dialog (`td-sync-worker-onboarding`) a direct, personal way to set up their account — a link, not "go register with your TD email and hope you spell it right."

**Non-Goals:**
- No live/cached Time Doctor roster check (rejected in the earlier design discussion — adds an external-API dependency to the auth-critical path; the pending-`Worker`-match check is sufficient and already exists).
- No retroactive action on `User` rows created before the toggle is flipped.
- No email-sending infrastructure — the claim link (see D4) is generated by the system but *delivered* manually by Admin (Slack, in person, etc.), same as the existing no-auto-invite decision in `td-sync-worker-onboarding`.
- Claim links are scoped to the "Add worker" dialog only (user decision, 2026-07-17) — not CSV-imported workers (they keep the existing self-register-by-email-match path; generating and distributing ~260 individual links is a different, larger problem nobody asked for) and not TD-sync-failure-resolve worker creation (same reasoning — could be revisited later if wanted).

## Integration Boundary

Auth-only change; no external integration (Time Doctor, Xero, Wise, n8n) is touched or newly depended on.

## Decisions

### D1 — Gate at the BetterAuth `before:create` hook, not the `/register` page or a separate middleware
`lib/auth.ts` already has a `before:create` hook (first-user-becomes-ADMIN). Adding the registration-open check there means it's enforced no matter which client calls `sign-up/email` (the React form, a script, a future mobile client) — a page-level check on `/register` alone would be bypassable by calling the API directly, which is exactly the gap `GET /api/profile`'s fix in the other proposal is closing for a different endpoint. Consistent enforcement point.

### D2 — New `AppConfig` singleton table, following the existing `TimeDoctorConfig`/`XeroToken` `id: "singleton"` pattern
No existing table is a natural fit for a general app-wide runtime flag. A one-row singleton table matches the pattern already established twice in this schema, rather than introducing a new pattern (e.g. an environment variable, which would require a deploy to change — defeating the purpose).

### D3 — Reject with a generic error message
The rejection response should not reveal *why* (e.g. "this email isn't recognized" vs. a generic "registration is currently closed") to avoid trivially confirming which emails are/aren't real contractors to an outside prober. A single generic message covers both the "registration closed" and "email not matched" cases.

### D4 — Claim link: a random token on `Worker`, redeemed at a public `/claim/[token]` page, composes with the registration gate for free
`Worker` gains `claimToken` (unique, random, e.g. 32 bytes base64url) and `claimTokenExpiresAt` (14–30 days), set only by the "Add worker" flow (`app/api/admin/workers/manual-add/route.ts`) right after `provisionWorker` succeeds — not folded into `provisionWorker`/`lib/worker-provisioning.ts` itself, since CSV import and TD-failure-resolve deliberately don't get one (see Non-Goals). `/claim/[token]` looks the worker up by token, checks `userId IS NULL` and `claimTokenExpiresAt > now()`, and if valid shows a name (editable) + password form. Submitting calls `auth.api.signUpEmail` server-side (not the public HTTP route) with the pending worker's `timeDoctorEmail`, which triggers the *same* existing `after:create` claim hook (`claimPreprovisionedWorker`) as normal registration — no new claiming logic.

The useful property this gives for free: **the claim-link route never needs to bypass the `registrationOpen` gate from D1.** The gate's own check ("is there an unclaimed `Worker` matching this email") is satisfied automatically, because the claim route only ever calls sign-up for an email it just confirmed has a valid pending `Worker` (that's what the token proved). No special-case flag, no "internal call, skip the gate" branch to get wrong.

Alternative considered: a shared secret / default password admin sets, worker changes on first login. Rejected in the earlier design discussion (see conversation history) — a shared secret across many pending accounts is a real credential-stuffing surface if even one account goes unclaimed for a while; a per-worker single-use token has no such shared-secret risk, and reuses BetterAuth's sign-up flow instead of needing a "force password change" mechanism BetterAuth doesn't have built in.

Token invalidation: the claim route's own `userId IS NULL` check is the real guard (idempotent — a second attempt with the same token after a successful claim just finds `userId` already set and rejects); `claimToken`/`claimTokenExpiresAt` are cleared on successful claim as hygiene, not as the primary defense.

## Risks / Trade-offs

- **[Risk] Admin flips the toggle before all pending workers have claimed** → some legitimate workers get locked out. Mitigation: show the unclaimed-pending-worker count directly next to the toggle so this is a visible, informed decision, not a guess; the toggle is instantly reversible (no deploy).
- **[Risk] Toggle state itself becomes stale/forgotten** (e.g. flipped off once, then a new legitimate need for open registration arises months later and nobody remembers the switch exists) → low severity, purely a UX/discoverability concern; the Admin Settings page is the natural, already-visited home for it.
- **[Risk] Claim link leaks** (forwarded to the wrong person, sits visible in a Slack channel) before the intended worker uses it → whoever has the link can claim that specific pending `Worker` row. Bounded blast radius (one specific worker slot, not account takeover of an existing user — `userId IS NULL` is required) and time-boxed by the 14–30 day expiry; mitigate further by admin regenerating the link (invalidating the old one) if a leak is suspected — see tasks.md for the regenerate action.
- **[Risk] Worker loses/never opens the link** → admin needs a "regenerate claim link" action on the worker detail page (new token + expiry, old one invalidated) rather than the worker being permanently stuck.

## Migration Plan

1. Add `AppConfig` table + migration (default `registrationOpen: true` — no behavior change on deploy).
2. Add the `before:create` hook check (inert while `registrationOpen: true`, i.e. safe to ship ahead of actually wanting to close it).
3. Add the Settings UI toggle + pending-worker count.
4. Add `Worker.claimToken`/`claimTokenExpiresAt`, the "Add worker" flow's token generation, the `/claim/[token]` page, and the regenerate action — independent of steps 1–3, can ship in either order.
5. Operationally: once the pending cohort is confirmed claimed (via the count shown in step 3), Admin flips the toggle — no further deploy needed.

**Rollback**: flip the toggle back to `true` — no code rollback needed for the common case; a genuine code rollback (reverting the hook check) is a normal revert if the hook logic itself has a bug.

## Open Questions

- Should the "unclaimed pending worker count" shown next to the toggle also break down by how long each has been pending (e.g. flag ones pending >90 days for Admin follow-up), or is a raw count sufficient for this change? Defaulting to a raw count; can be revisited if Admin finds it insufficient in practice.
