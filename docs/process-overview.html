⚙️ Wise Payment Approach — Requires API Spike

Wise will be integrated directly from Portal, not through n8n. The exact payment mode must be verified against the real Wise Business account first, because API-created transfers may not support the same approval behavior as dashboard-created transfers.

Mode A — Manual Payment Record (Safe Baseline)
  • 🏦 Lorena/Felipe pays externally in Wise/Revolut/Crypto/PayPal
  • 🧾 Finance records payment details and receipt in Portal
  • ✅ Portal marks invoice PAID only after a local payment record exists
  • ✅ Safest first step while Wise API capabilities are verified
  • ✅ Works for developers and payment methods Wise cannot automate
Mode B — Portal Direct Wise API
  • 🏦 Portal creates quotes, recipients, transfers, and funding directly
  • 📦 Batch Groups must be evaluated before one-transfer-per-worker loops
  • 🔔 Wise webhook hits Portal directly; n8n only receives notifications after Portal saves state
  • ⚠️ Requires account/auth verification for funding and approval behavior
  • ⚠️ Should start only after a Wise sandbox/business-account spike
Wise spike required — Verify sandbox URL, auth model, API funding, Batch Groups, Business Payment Approvals behavior, and webhook signature verification before implementing automatic payments.

1. Double-Payment Prevention

Every Wise transfer is created with a unique customerTransactionId equal to the invoice's database ID. Wise guarantees idempotency — sending the same ID twice never results in a second charge.

customerTransactionId = invoice.id (UUID)
DB unique constraint: workerId + period + source
Prevents: Duplicate Payment
🔐

2. Wise API Key Scope Limitation

The Wise credential stored for Portal must use the least privilege model supported by the actual business account. Personal-token, OAuth, funding, and approval limitations must be verified before production automation is enabled.

Verify: auth model · transfer permissions · balance/funding access · approval behavior
Store production credential outside DB unless a future design explicitly approves encrypted storage
Prevents: Unauthorized Account Drain
✅

3. Wise Webhook Signature Verification

When Wise calls our webhook to confirm a payment, Portal verifies Wise's signature before updating any invoice status. An attacker cannot fake a "payment sent" notification to mark invoices as paid without actual money moving.

X-Signature-SHA256 + Wise public key verification
Reject any request with invalid signature → 401
Prevents: Fake Payment Confirmation
🔑

4. Credential Storage Strategy

API credentials are stored with different security levels based on their privilege:

Xero OAuth tokens → Database (rotated automatically)
Time Doctor token → Database (read-only, 6-month TTL)
Wise API key → Environment variable only
   Never stored in DB. Access requires server restart to change.
Prevents: Key Exfiltration via DB Access
👤

5. Mandatory Human Approval Gate

No money moves without Lorena (or an Admin) explicitly reviewing the monthly payment summary and clicking Approve Batch. This step cannot be bypassed by the scheduler. The platform can generate invoices automatically, but payment execution always requires a deliberate human action.

Cron → auto-generate invoices only
Payment → requires Admin session + explicit POST
Prevents: Unintended/Incorrect Payments
🚨

6. Anomaly Detection & Alerts

Before presenting the payment summary to Lorena, the platform compares each worker's current month hours against their 3-month rolling average. Workers with >30% variation are highlighted with a warning — Lorena can review and override before payment proceeds.

>30% vs avg → 🟡 Review flag
>100% vs avg → 🔴 Block, require manual override
New worker (no history) → 🔵 First payment flag
Prevents: Overpayment from Data Errors
📋

7. Full Audit Trail (Payment Runs)

Every payment batch is recorded as a Payment Run — who triggered it, when, how many invoices, total amounts, which succeeded, which failed, and why. This log is immutable and stored independently from the invoices themselves.

PaymentRun: id · period · triggeredBy
  · status · totalAmount · processedAt
PaymentRunItem: invoiceId · wiseTransferId · status
Enables: Reconciliation & Dispute Resolution
⚙️

8. Dual Payment Mode — Configurable Safety Level

The platform keeps manual payment recording as the safe baseline while Wise API capabilities are verified. API-created transfers, Batch Groups, funding, and approval behavior must be tested against the real Wise Business account before automatic payments are enabled.

Baseline: External payment → Portal payment record → PAID
Future: Portal Wise API → Wise webhook → verified Portal update
n8n: notification-only after Portal state is saved.
Enables: Segregation of Duties
⏰

9. Cron Endpoint Protection

The scheduled sync endpoint (/api/admin/sync/timedoctor) is protected by a CRON_SECRET header that only Vercel's scheduler knows. Even if the URL is discovered, it cannot be triggered externally without the secret.

Vercel adds: Authorization: Bearer {CRON_SECRET}
Platform validates header before processing
Returns 401 to all other callers
Prevents: Unauthorized Sync Triggering
